Project 1
Automated Enterprise ETL Reverse Engineering at Scale

1.1 Executive Summary

Designed and built a production-grade data lineage analysis platform that automatically reverse-engineered an undocumented enterprise ETL landscape spanning ~4,300 mappings, ~1,800 tables, and 20+ source systems — a task that would have taken a team of analysts months to do manually.

The platform performs deep, multi-tool lineage tracing across Oracle Data Integrator (ODI), Oracle Job Chains (OJC), IBM DataStage, PL/SQL stored procedures, and PostgreSQL Data Lake landing zones — producing interactive HTML reports with full SQL visibility at both column-level and table-level granularity. It was built from scratch in Python to solve a critical gap at a large financial institution: thousands of ETL jobs across multiple orchestration tools with zero end-to-end documentation of actual data flows and transformations.

1.2 Problem Statement

The client's data warehouse environment consisted of:

  • Oracle Data Integrator (ODI) as the primary ETL engine with ~4,300 mappings across ~20 projects and ~120 folders
  • IBM DataStage jobs handling a significant portion of ETL processing, particularly for legacy data integration and complex transformation pipelines
  • PL/SQL scripts and stored procedures implementing substantial business logic in Oracle databases, including validation routines, partition management, and statistics gathering (DBMS_STATS)
  • Oracle Job Chains (OJC) orchestrating multi-stage screening pipelines (Staging → Data Quality → Core → Views → Targets)
  • Groovy scripts embedded within ODI mapping expressions and transformations for complex data manipulation
  • PostgreSQL Data Lake with ~2 TB across ~1,800 tables in ~60 landing zone schemas
  • Oracle DWH with multiple schemas: Core (~340 tables), Reporting (~220 target + ~190 regulatory + ~250 view objects), Analytics (~1,700 tables)
  • 20+ upstream source systems including Core Banking (~300 tables), Regulatory Reporting (~240 tables), Digital Banking (~100 tables), AML/Compliance, Customer Management, Data Privacy, Risk Management, and others

No single tool or document captured the end-to-end data flow from source systems through landing zones, staging, data quality, core layer, views, and final reporting targets.

1.5 Key Capabilities

1.5.1 Column-Level Lineage

  • Traces a single column from any target/regulatory table back to its original source(s)
  • Full DDL/SQL retrieval from ODI session logs
  • Expression parsing through all transformation layers (including Groovy expressions)
  • Terminal source identification (Data Lake landing zone tables)
  • Cycle detection with path recording
  • Source origin badges: [SESSION LOG ✓], [VIEW DDL], [OJC], [GENERATED]
  • Maximum recursion depth: configurable (default 15 levels)

1.5.2 Table-Level Lineage

  • Identifies all source tables feeding into a target table
  • Classifies sources into categories: Data Lake, External, Unmapped, cross-target sources
  • Full lineage graph traversal with visited-node tracking (prevents cycles)
  • Produces structured JSON output for downstream automation
  • Exports: edges, SQL steps, build sequence (topological sort), and ODI/OJC mapping metadata

1.5.3 Batch Processing

  • Automated processing of all ~220 target tables from the Reporting schema
  • Progress tracking with resume capability for interrupted runs
  • Per-table HTML reports + CSV/XLSX summary with statistics
  • Performance: 30–120 seconds per table, full batch in ~2–4 hours

1.5.4 Analytics & Dashboards

  • Tableau-ready SQL generation for business dashboards
  • Dimension data export to CSV for analytics platforms
  • DWH dimension inspection tools for data governance

Project 2
Legacy-to-Lakehouse Platform Migration — Oracle to Spark + Delta Lake

2.1 Executive Summary

Architected and implemented a full-scale migration framework and process for an enterprise data warehouse — from a dual-database Oracle + PostgreSQL architecture to a modern open-source Data Lakehouse. The migration pipeline, patterns, and tooling were successfully established and handed over to the data engineering team for ongoing execution — replacing millions of dollars in licensed tooling with a fully version-controlled, testable, and reproducible platform.

The target architecture uses dbt (data build tool) with a Medallion pattern (Bronze → Silver → Gold), Apache Spark for compute, MinIO (S3-compatible) for storage, Delta Lake for ACID transactions, Hive Metastore for schema management, Kyuubi as the JDBC gateway, and Apache Airflow for orchestration.

Project 3
AI Multi-Agent Orchestration — Autonomous Migration Pipeline

3.1 Executive Summary

Designed and built a custom AI orchestration engine that turns what would be months of manual data engineering into an autonomous, continuously running migration pipeline — coordinating a network of specialized AI agents that extract, translate, build, test, and deploy dbt models with minimal to zero human supervision.

Instead of a data engineer manually converting each legacy SQL model one at a time, this system uses a state machine-based orchestration engine that dispatches work to purpose-built AI agents (lineage extraction, SQL translation, model compilation, cross-platform verification, diagnosis). The agents collaborate through a shared PostgreSQL knowledge base, processing models in parallel and continuously — achieving migration throughput that is orders of magnitude faster than any manual or single-agent approach.

Key Architectural Principle: Existing Chaos Migration

This approach embodies an "existing chaos migration" principle: dbt can connect to virtually any source and target system (30+ database adapters), meaning the same methodology works regardless of the starting point (Oracle, SQL Server, Teradata, DB2, Informatica, DataStage, SSIS) or the destination (Spark, Snowflake, Databricks, BigQuery, PostgreSQL, DuckDB).

3.3.2 AI-Powered Testing (Where dbt Cannot Reach)

  1. Generate verification queries dynamically based on KB metadata
  2. Execute against both local dbt output and production Oracle/PostgreSQL
  3. Compare results with configurable tolerance for numeric precision
  4. Report discrepancies with root cause hypothesis
  5. Distinguish expected differences (e.g., date handling design decisions)

3.3.3 Global Error Policy

The orchestration implements a strict fail-fast policy:

  1. Error occurs → document in report file
  2. Attempt ONE automated fix based on error pattern
  3. Re-run the failed step
  4. If STILL fails → STOP IMMEDIATELY, escalate to human
  5. No looping, no alternative approaches without human approval

3.4 Prototyping Strategy — Open Source First

  1. Start with open-source/local databases — PostgreSQL, DuckDB, or SQLite as both source and target
  2. Run the full pipeline end-to-end locally — extraction, KB loading, model generation, compilation, testing
  3. Validate the AI orchestration works before incurring any cloud costs
  4. Scale to cloud only after local verification — deploy to Spark/MinIO/Delta Lake or Snowflake/Databricks
